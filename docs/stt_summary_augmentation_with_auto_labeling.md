좋습니다. 이 주제는 단순히 “라벨을 자동으로 붙인다” 수준이 아니라,
**STT 전사원문 → 요약정답 → 핵심키워드**를 하나의 학습/평가 자산으로 보고,
**Auto Labeling으로 정답 데이터셋의 양·일관성·재사용성**을 높이는 설계가 핵심입니다.

데이터과학자 관점에서는 아래처럼 봐야 합니다.

---

# 1. 목표 정의부터 분리해야 합니다

질문의 핵심 데이터는 3개입니다.

* **STT 전사원문**

  * 상담사/고객 대화 전체
  * 비정형, 노이즈 많음, 화자 분리 이슈 존재
* **요약정답**

  * 사람이 만든 이상적 요약문
  * 학습용 gold label 또는 weak gold
* **핵심키워드**

  * 요약의 핵심 슬롯
  * 예: 상품명, 요청사항, 처리결과, 민원유형, 감정, 이슈 키워드

여기서 Auto Labeling은 보통 아래 4가지 목적에 씁니다.

### ① 요약 학습용 라벨 확장

* 원래 사람이 만든 요약정답이 적을 때
* STT 원문에서 준정답 요약을 자동 생성

### ② 키워드 추출 라벨 확장

* 요약정답/원문에서 핵심 엔티티, 행위, 결과를 자동 태깅

### ③ 품질평가용 라벨 생성

* 누락/과다/오답/정답 판정용 근거 라벨 생성

### ④ 라우팅/분류용 보조 라벨 생성

* 일반문의 / 민원 / 해지 / 보험금청구 / 주소변경 등 intent 자동 태깅

즉,
**Auto labeling은 “요약문 하나 생성”이 아니라, 다층 라벨 체계를 만드는 작업**입니다.

---

# 2. 전체 기술 구조: Gold / Silver / Weak Label 체계로 가야 합니다

실무에서는 전량을 사람이 라벨링할 수 없기 때문에
다음 구조를 권장합니다.

## 2-1. 라벨 등급 체계

### Gold Label

* 현업/QA가 검수 완료
* 가장 신뢰도 높음
* 학습/평가 기준셋

### Silver Label

* Auto labeling 결과 + 룰 검증 + 일부 샘플검수 통과
* 대규모 학습에 사용 가능

### Weak Label

* LLM, 규칙, 사전 기반으로 만든 초안
* 직접 학습에 넣기보다 필터링 후 사용

---

# 3. 권장 데이터 스키마 설계

Auto labeling 이전에 스키마를 먼저 통일해야 합니다.

## 3-1. 원천 단위

한 건의 통화/채팅 단위를 아래처럼 잡습니다.

```json
{
  "call_id": "C000123",
  "channel": "콜센터",
  "stt_transcript": "...",
  "utterances": [
    {"speaker": "고객", "text": "...", "start": 0.1, "end": 3.2},
    {"speaker": "상담사", "text": "...", "start": 3.3, "end": 7.1}
  ],
  "human_summary": "...",
  "human_keywords": ["주소변경", "자동이체", "보험료"],
  "metadata": {
    "product": "...",
    "date": "...",
    "agent_id": "..."
  }
}
```

## 3-2. Auto labeling 결과 스키마

```json
{
  "call_id": "C000123",
  "auto_labels": {
    "summary_label": {
      "text": "...",
      "method": "llm_v1",
      "confidence": 0.86
    },
    "keyword_labels": [
      {"keyword": "주소변경", "type": "intent", "confidence": 0.95},
      {"keyword": "자동이체", "type": "service", "confidence": 0.91}
    ],
    "issue_type": {
      "label": "계약변경",
      "confidence": 0.88
    },
    "quality_tags": {
      "missing_slots": [],
      "hallucination_risk": false
    }
  }
}
```

핵심은
**라벨값만 저장하지 말고 생성 방법(method), confidence, 검증결과를 같이 저장**하는 것입니다.

---

# 4. Auto Labeling 적용 방식: 4단 하이브리드가 가장 안정적입니다

단독으로 LLM만 쓰면 흔들립니다.
권장 순서는 아래입니다.

## 4-1. 1단계: 규칙 기반 전처리 라벨링

가장 먼저 해야 합니다.

대상:

* 상담 채널 분류
* 화자 분리 정합성
* 전화번호, 주민번호, 계좌번호 등 PII 마스킹
* 상품명 사전 매핑
* 자주 나오는 업무코드/서비스 코드 태깅

예:

* “주소를 바꾸고 싶어요” → `intent_candidate=주소변경`
* “자동이체 계좌 바꿔주세요” → `intent_candidate=자동이체변경`

이 단계의 목적은
**LLM이 불안정하게 추론하지 않도록 후보 라벨을 먼저 좁히는 것**입니다.

---

## 4-2. 2단계: 사전/온톨로지 기반 엔티티 라벨링

핵심키워드는 그냥 추출하면 안 되고,
**도메인 사전과 연결된 정규화 키워드**여야 합니다.

예:

* “이체계좌 바꾸기”, “자동이체 계좌변경”, “출금계좌 바꿔줘”
  → 모두 `자동이체변경`

### 적용 기술

* Dictionary matching
* Synonym expansion
* 형태소 기반 명사구 추출
* Fuzzy match
* 약어/오타 정규화

### 추천 라벨 구조

* surface form: 원문 표현
* canonical label: 표준 키워드
* entity type: 상품/행위/사유/결과/감정 등

예:

```json
{
  "surface": "출금계좌 바꾸기",
  "canonical": "자동이체변경",
  "entity_type": "intent"
}
```

---

## 4-3. 3단계: LLM 기반 요약/키워드 Auto Labeling

여기서 본격적으로 요약정답과 핵심키워드를 생성합니다.

### LLM이 맡아야 하는 역할

* 긴 STT에서 핵심 이벤트 압축
* 고객요청 / 상담사안내 / 처리결과 분리
* 명시적 키워드 외에 잠재 핵심 이슈 추론
* 다중질문 구조 분해

### 가장 중요한 포인트

요약과 키워드를 한 번에 자유 생성시키지 말고
**구조화 출력(JSON)** 으로 강제해야 합니다.

예:

```json
{
  "summary": {
    "customer_intent": "...",
    "agent_action": "...",
    "result": "..."
  },
  "keywords": [
    {"term": "주소변경", "type": "intent"},
    {"term": "보험료", "type": "domain"}
  ],
  "slots": {
    "product_name": "...",
    "request_type": "...",
    "status": "완료"
  }
}
```

### 프롬프트 핵심 원칙

* 원문에 없는 내용 추정 금지
* 상담사/고객 발화를 구분
* 결과 미확정이면 “미확인” 처리
* 핵심키워드는 표준어 기준으로 정규화
* 출력 필드 고정

---

## 4-4. 4단계: Judge / Validator 모델로 후검증

Auto labeling에서 가장 중요한 단계입니다.

생성 모델과 검증 모델을 분리해야 합니다.

### 검증 항목

* 요약이 원문과 일치하는가
* 키워드가 원문 근거를 가지는가
* 누락된 핵심 요청이 있는가
* 환각성 정보가 있는가
* 요약정답 포맷을 만족하는가

### 방법

* LLM Judge
* Rule validator
* Embedding similarity
* Slot consistency check

예:

* 요약문에 “자동이체 변경 완료”가 있는데 원문에 완료 근거가 없으면 reject
* 키워드 “해지”가 있는데 실제로는 “해지 문의만” 한 경우 별도 태그

---

# 5. STT 전사원문 기준 Auto Labeling 세부 방법

STT는 노이즈가 많아서 바로 요약하면 정확도가 떨어집니다.
먼저 **전사 정제 라벨링**을 해야 합니다.

## 5-1. 전처리

* 불필요 반복 제거
* 필러어 제거: “아”, “그”, “어”, “음”
* STT 오인식 보정 사전 적용
* 화자 오류 수정
* 문장 경계 복원
* 숫자/상품명 정규화

예:

* “자동이체가 계좌가 좀 바꿔고 싶거든요”
  → 의미 정제 후 intent 후보 추출

## 5-2. 발화 단위 라벨링

대화 전체가 아니라 utterance 단위로 먼저 태깅하면 품질이 올라갑니다.

### utterance label 예시

* 고객요청
* 상담사확인질문
* 본인인증
* 처리안내
* 결과통보
* 감정표현
* 불만제기

이렇게 발화 기능 라벨이 있으면
최종 요약 생성 때 핵심 발화만 모아서 쓰기 쉬워집니다.

---

# 6. 요약정답 Auto Labeling 방법

요약정답은 그냥 “짧게 줄인 문장”이 아닙니다.
보험/상담 데이터에서는 보통 아래 3요소를 갖춰야 합니다.

* 고객 요청
* 상담사 처리/안내
* 최종 상태

## 6-1. 추천 방식: 단계형 생성

### Step 1. 핵심 발화 추출

* 고객의 주요 요구 발화
* 상담사의 실제 조치 발화
* 결과 발화

### Step 2. 슬롯 추출

* 요청유형
* 대상상품
* 변경항목
* 처리상태
* 추가문의

### Step 3. 요약문 생성

슬롯 기반으로 정답 요약 포맷에 맞춰 생성

예:

```json
{
  "request_type": "주소변경",
  "target_product": "종신보험",
  "agent_action": "변경 절차 안내",
  "result": "고객이 모바일 앱에서 변경 진행 예정"
}
```

이를 다시 자연어로 변환:

* `고객은 종신보험 주소변경을 요청했고, 상담사는 모바일 앱을 통한 변경 절차를 안내했으며 고객은 추후 직접 진행하기로 했다.`

이 방식이 자유 생성보다 훨씬 안정적입니다.

---

## 6-2. Teacher-Student 방식

사람이 만든 요약정답 일부가 있다면 가장 효과적입니다.

### 방법

* Gold 샘플로 강한 LLM prompt 설계
* LLM이 Silver summary 생성
* 사람이 일부 검수
* 고품질 Silver를 student model 학습에 사용

즉,
**사람 요약정답 → LLM 모방 생성 → 검증 → 재학습** 구조입니다.

---

# 7. 핵심키워드 Auto Labeling 방법

핵심키워드는 보통 3층 구조가 가장 좋습니다.

## 7-1. 키워드 타입 분리

* Intent keyword: 주소변경, 자동이체변경, 해지문의
* Domain keyword: 보험료, 계약자, 피보험자
* Entity keyword: 상품명, 특약명, 수술명
* Outcome keyword: 처리완료, 미처리, 재문의예정
* Emotion keyword: 불만, 긴급, 민원우려

## 7-2. 추출 방식

### 규칙 기반

* 상품명, 코드, 조직명
* 고정 업무용어

### 모델 기반

* KeyBERT류 추출
* sequence labeling
* LLM structured extraction

### 요약 기반 보강

* 요약문에서 다시 키워드 추출
* 원문+요약 동시 기준으로 교집합/합집합 전략

---

## 7-3. 추천 저장 방식

단순 배열보다 아래처럼 저장하는 것이 좋습니다.

```json
[
  {
    "keyword": "주소변경",
    "canonical": "주소변경",
    "type": "intent",
    "source": "llm+rule",
    "confidence": 0.94,
    "evidence_span": "주소를 바꾸고 싶어요"
  }
]
```

핵심은 **evidence span**입니다.
나중에 품질검증과 현업설명 가능성이 올라갑니다.

---

# 8. 데이터 증강 관점의 Auto Labeling 전략

Auto labeling은 단지 라벨 부착이 아니라
**학습 가능한 샘플 수를 확장하는 데이터 증강**이어야 합니다.

## 8-1. 증강 방식 1: 동일 원문에 다중 라벨 생성

한 전사원문에서 여러 자산 생성

* 요약정답
* 핵심키워드
* intent
* 감정
* 처리결과
* 상담단계

즉, 1건의 대화가 다중 task 학습데이터가 됩니다.

---

## 8-2. 증강 방식 2: 요약 표현 다양화

같은 의미의 요약을 여러 스타일로 생성

예:

* 짧은 요약
* 표준 요약
* 슬롯 기반 요약
* 현업 상담일지 스타일 요약

이렇게 하면 요약 모델이 특정 문장패턴에 과적합되는 걸 줄일 수 있습니다.

단, 의미 동일성 검증이 꼭 필요합니다.

---

## 8-3. 증강 방식 3: 키워드 변형 확장

표준 키워드 하나에 대해

* 동의어
* 오타형
* 구어체형
* 축약형

을 연결해 retrieval/분류 성능을 높일 수 있습니다.

예:

* `자동이체변경`

  * 자동이체 계좌변경
  * 출금계좌변경
  * 자동납부 계좌 바꾸기

---

## 8-4. 증강 방식 4: Hard Negative 생성

학습/평가에서 매우 중요합니다.

예:

* 비슷하지만 다른 intent

  * 주소변경 vs 계약자변경
  * 자동이체변경 vs 납입일변경
* 비슷한 키워드인데 실제로는 언급만 한 경우

  * 해지 문의 vs 해지 신청 완료

이런 hard negative를 자동 생성하면
분류/추출/요약 정확도가 좋아집니다.

---

# 9. 권장 파이프라인

데이터과학자 관점에서 전체 파이프라인은 이렇게 설계하는 게 좋습니다.

## 9-1. Offline labeling pipeline

1. STT 정제
2. 발화 분리 및 화자 보정
3. 규칙 기반 초기 태깅
4. 사전 기반 엔티티 라벨링
5. LLM 구조화 요약/키워드 생성
6. Judge 검증
7. confidence scoring
8. Gold/Silver/Weak 분류 저장

## 9-2. Human-in-the-loop

* confidence 낮은 샘플 우선 검수
* 신규 intent/신규 상품 위주 검수
* 오탐 패턴 수집 후 룰 업데이트

## 9-3. Active Learning

* 모델 불확실도가 높은 샘플
* 사람과 모델 불일치 샘플
* 신규 도메인 샘플

을 우선적으로 재라벨링

---

# 10. 기술별 권장 역할 분담

## 규칙/사전이 잘하는 것

* 코드/상품명/고정업무용어
* 숫자/날짜/계좌/PII
* 정규화/표준화
* 반복적 intent 후보

## LLM이 잘하는 것

* 긴 대화 요약
* 다중 의도 분해
* 암묵적 요청 추론
* 문맥 기반 핵심어 추출

## Judge 모델이 잘하는 것

* 요약의 근거성 검증
* 키워드 누락 탐지
* 환각 탐지
* 정답 포맷 일관성 확인

# 11. 품질관리 지표

Auto labeling은 반드시 측정해야 합니다.

## 요약 라벨 품질

* Faithfulness
* Coverage
* Conciseness
* Slot accuracy
* Human acceptance rate

## 키워드 라벨 품질

* Precision / Recall / F1
* Canonical mapping accuracy
* Evidence span accuracy
* Type classification accuracy

## 운영 지표

* Auto labeling acceptance rate
* Human correction rate
* Label drift rate
* 신규 패턴 발견율

# 12. 가장 추천하는 실전 적용 방식

보험/상담 데이터 기준으로 가장 현실적인 방법은 아래 조합입니다.

## 추천안

### 1단계

* STT 정제
* 화자 분리
* PII 마스킹
* 도메인 사전 구축

### 2단계

* 규칙 기반 intent/엔티티 초기 태깅
* 상품명/업무유형/처리상태 슬롯화

### 3단계

* LLM으로 구조화 요약 생성
* LLM으로 핵심키워드 생성
* 출력은 JSON 강제

### 4단계

* LLM Judge + Rule validator 검증
* confidence score 산출

### 5단계

* Gold / Silver / Weak 분리 저장
* Silver만 대규모 학습에 투입
* Gold는 평가셋과 prompt calibration용 유지

**학습 가능한 Silver dataset을 대량 생산하는 데이터 엔지니어링 + 데이터과학 작업**으로 설계해야 합니다.
